# CONVENTIONS.md — Coding Standards & Patterns

This document defines the coding conventions for OrganizaTUM. All code — whether written by hand or generated by AI — must follow these rules.

---

## File Naming

| Type                  | Pattern             | Example              |
| --------------------- | ------------------- | -------------------- |
| React components      | `PascalCase.tsx`    | `ChatPanel.tsx`      |
| Hooks                 | `camelCase.ts`      | `useChat.ts`         |
| Utilities / libs      | `kebab-case.ts`     | `api-client.ts`      |
| Zod schemas           | `kebab-case.ts`     | `calendar.ts`        |
| Prompt templates      | `kebab-case.ts`     | `onboarding.ts`      |
| Constants / config    | `kebab-case.ts`     | `model-config.ts`    |
| Test files            | `*.test.ts(x)`      | `ChatPanel.test.tsx` |
| Types (inferred only) | same file as schema | inside `calendar.ts` |

## TypeScript Rules

### Strict Mode

`strict: true` in every `tsconfig.json`. No exceptions.

### No `any`

Never use `any`. Options when you don't know the type:

1. Define a Zod schema and infer (preferred)
2. Use `unknown` and narrow with type guards
3. Use a generic

```typescript
// WRONG
function process(data: any) { ... }

// CORRECT
function process(data: unknown) {
  const parsed = SomeSchema.safeParse(data);
  if (!parsed.success) throw new ValidationError(parsed.error);
  // parsed.data is fully typed
}
```

### Imports

Use `import type` for type-only imports:

```typescript
// CORRECT
import type { TimeBlock } from "@organizaTUM/shared";
import { TimeBlockSchema } from "@organizaTUM/shared";

// WRONG
import { TimeBlock, TimeBlockSchema } from "@organizaTUM/shared";
```

### Import Order

Enforce this order with a blank line between groups:

```typescript
// 1. Node built-ins
import { readFile } from "fs/promises";

// 2. External packages
import { z } from "zod";
import { useChat } from "ai/react";

// 3. Workspace packages
import { TimeBlockSchema } from "@organizaTUM/shared";

// 4. Relative imports
import { ChatInput } from "./components/ChatInput";
```

### Exports

Use named exports everywhere:

```typescript
// CORRECT
export function ChatPanel() { ... }
export const TimeBlockSchema = z.object({ ... });

// WRONG
export default function ChatPanel() { ... }
```

Exception: Next.js pages and layouts require default exports — this is the only allowed case.

## React Conventions

### Component Structure

```typescript
// 1. Imports
import { useState } from 'react';
import type { TimeBlock } from '@organizaTUM/shared';

// 2. Types (component-local only)
interface ChatPanelProps {
  initialMessages?: Message[];
}

// 3. Component
export function ChatPanel({ initialMessages = [] }: ChatPanelProps) {
  // hooks first
  const [isLoading, setIsLoading] = useState(false);

  // handlers
  const handleSubmit = () => { ... };

  // render
  return ( ... );
}
```

### Hooks

- Custom hooks go in `features/<feature>/hooks/`.
- Hook files export one hook. File name matches hook name.
- Hooks always start with `use`.

### State Management

- **Local UI state** → `useState` / `useReducer`
- **Global app state** (user profile, calendar) → Zustand store in `src/stores/`
- **Server state** (chat messages, streaming) → Vercel AI SDK hooks

Do NOT mix these. A Zustand store should never manage streaming state. Vercel AI SDK should never manage user preferences.

### Conditional Rendering

Always handle all states explicitly:

```typescript
// CORRECT — all states handled
if (isLoading) return <SkeletonLoader />;
if (error) return <ErrorDisplay message={error} />;
if (!data || data.length === 0) return <EmptyState />;
return <DataView data={data} />;

// WRONG — only handles success
return <DataView data={data} />;
```

## Zod Schema Conventions

### Location

All schemas live in `packages/shared/src/schemas/`. No schema definitions in `apps/`.

### Naming

```typescript
// Schema: PascalCase + "Schema" suffix
export const TimeBlockSchema = z.object({ ... });

// Inferred type: PascalCase, no suffix
export type TimeBlock = z.infer<typeof TimeBlockSchema>;
```

### Validation Pattern (Backend)

```typescript
export function validateAgentResponse<T>(
  schema: z.ZodSchema<T>,
  data: unknown,
): T {
  const result = schema.safeParse(data);
  if (!result.success) {
    console.error("Agent response validation failed:", result.error.flatten());
    throw new AgentValidationError(result.error);
  }
  return result.data;
}
```

Always use `safeParse`, never `parse` in production paths. We want to handle errors gracefully, not crash.

## Agent Prompt Conventions

### File Structure

Each prompt file exports:

1. A `SYSTEM` constant — the static system prompt
2. Builder functions — dynamic prompt generators

```typescript
// prompts/scheduling.ts

export const SCHEDULING_SYSTEM = `
You are a scheduling optimizer for university students.
You MUST respond ONLY in valid JSON matching the provided schema.
Never include markdown, preamble, or text outside the JSON object.
`;

export function buildSchedulingPrompt(
  profile: UserProfile,
  analysis: CourseAnalysis[],
  constraints: TimeBlock[],
): string {
  return `
Student Profile:
${JSON.stringify(profile, null, 2)}

Course Analysis:
${JSON.stringify(analysis, null, 2)}

Fixed Constraints (do not overlap):
${JSON.stringify(constraints, null, 2)}

Generate a weekly calendar. Rules:
- Harder courses get more study time
- Learning style: ${profile.learningStyle}
- Include meal breaks and leisure blocks
- No block before 08:00 or after 22:00

Return a JSON object matching the WeeklyCalendar schema.
  `.trim();
}
```

### Prompt Rules

- Never inline prompts in agent node logic. Always import from `prompts/`.
- Always include the structured output instruction in system prompts.
- Include the Zod schema description or a JSON example in the prompt so the LLM knows the exact shape.
- Dynamic data is injected via template functions, never string concatenation in the node.

## Error Handling

### Backend

```typescript
// Use a consistent error response shape
interface ApiError {
  error: string; // machine-readable code
  message: string; // human-readable message
  details?: unknown; // optional debug info (omit in production)
}

// Example
return Response.json(
  {
    error: "VALIDATION_FAILED",
    message: "The generated schedule has overlapping blocks.",
  },
  { status: 422 },
);
```

### Frontend

- Network errors → toast notification + retry option
- Validation errors → display inline error message
- Agent errors → show in chat as a system message: "I had trouble generating that. Let me try again."
- Never show raw error objects or stack traces to the user.

## Git Conventions

### Branch Naming

```
feature/<short-description>    # feature/chat-panel
fix/<short-description>        # fix/calendar-overlap
```

### Commit Messages

```
feat: add onboarding conversation flow
fix: resolve calendar block overlap on Wednesdays
refactor: extract prompt templates to dedicated files
docs: update ARCHITECTURE.md with refinement node spec
```

Prefix with `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`, or `style:`.

## Tailwind Conventions

- Use utility classes directly. No `@apply` in CSS files.
- For repeated patterns, extract a React component — not a CSS class.
- Color palette: stick to Tailwind defaults for the hackathon. Don't create custom theme colors unless branding is required.
- Responsive: mobile-first. Use `md:` breakpoint for desktop split-panel layout.

```tsx
// CORRECT — component extraction
function StatusBadge({ status }: { status: string }) {
  return <span className="rounded-full bg-green-100 px-2 py-1 text-xs text-green-800">{status}</span>;
}

// WRONG — @apply in CSS
.status-badge { @apply rounded-full bg-green-100 px-2 py-1 text-xs text-green-800; }
```
